## 一、项目背景与问题定义

### 1.1 业务痛点

商业综合体（如万达广场）的运营团队每天需要制作"经营日报"，包含：
- 广场客流、进店客流、停车场车流与收入
- 超市客流、影院客流及其占比
- 广场销售额
- 以上指标的**日环比、周环比、年同比**
- 以上指标的**月累计、月日均、月环比、月同比**
- 以上指标的**年累计、年日均、周末日均、平日日均**

人工整理一份完整简报需要 **30-60 分钟**，且极易出错（复制粘贴错误、公式拖拽错误等）。

### 1.2 项目目标

**用一套 Python 程序，全自动完成以下流程：**

```
API 数据拉取 → 多维度计算 → 精美报表渲染 → 截图成长图 → 推送飞书群
```

**核心指标：** 从手工 30 分钟 → 自动 15 秒，零人工干预。

## 二、技术选型复盘

### 2.1 为什么选 Python + HTML 而不是纯 Excel？

| 维度 | Excel/VBA | Python + HTML |
|------|-----------|---------------|
| 自动化程度 | 需要手动触发宏 | 完全无人值守 |
| 视觉表现力 | 局限于 Excel 图表 | ECharts 专业级可视化 |
| 推送能力 | 无法直接推送 | 飞书 API 自动推送 |
| 可维护性 | VBA 调试困难 | Python 模块化清晰 |
| 复用性 | 难以移植 | 改配置即可复用 |

**结论：** Python + HTML 模板方案在自动化、可视化、可维护性上全面胜出。

### 2.2 为什么选 Jinja2 + ECharts 而不是前端框架？

- **零构建成本**：不需要 Node.js、Webpack、npm，纯 Python 生态
- **静态渲染**：报表是一次性生成的静态内容，不需要交互式前端框架
- **ECharts CDN**：通过 CDN 引入，一行代码解决图表库依赖
- **Html2Image**：直接将渲染好的 HTML 截图为 PNG，不需要浏览器交互

### 2.3 为什么用 configparser 而不是 YAML/JSON？

- **用户友好**：运营人员不懂 YAML 缩进规则，`config.ini` 更直观
- **热修改**：直接用记事本打开就改，不需要任何工具
- **中文注释**：每个参数旁边都有中文注释，自解释

## 三、架构设计复盘

### 3.1 三层架构

```
配置层 (config.ini)
    ↓ 读取配置
计算层 (run.py: ReportEngine)
    ↓ 传递数据
展示层 (dynamic_template.html: Jinja2 + ECharts)
```

**这种分层的好处：**
- 换广场？→ 改配置层
- 换指标？→ 改计算层
- 换样式？→ 改展示层
- 三层互不影响

### 3.2 ReportEngine 的设计哲学

`ReportEngine` 是整个系统的核心，其设计遵循以下原则：

**原则一：一个方法走天下**

`build_row()` 是唯一的行数据构建入口。不管是客流、销售、还是进店率，所有指标都走这一个方法。通过参数组合实现差异化：

```python
# 客流（绝对值，除以10000显示为"万"）
build_row("广场客流(万)", "passengerFlowValue", None, 10000, False, "num")

# 进店率（比率，分子/分母）
build_row("进店率", "storeFlowValue", "passengerFlowValue", 1, False, "ratio")

# 车流（整数，不除）
build_row("出场车流(辆)", "car_count", None, 1, True, "int")
```

**原则二：区间查询而非逐日累加**

`get_stats(start, end, key)` 使用 Pandas 的向量化筛选，避免了手写循环：

```python
mask = (df['dt'] >= start) & (df['dt'] <= end)
sub = df[mask]
return sub[key].sum(), sub.shape[0]
```

**原则三：日期驱动一切**

所有的比较维度（日环比、周环比、年同比、月累计、年累计）都通过日期偏移来定义：

```python
p = {
    "n": (d, d),                    # 本日
    "y": (d-1天, d-1天),             # 昨日
    "w": (d-7天, d-7天),             # 上周同日
    "a": (d-364天, d-364天),          # 去年同期
    "mtd": (月初, d),                # 月累计
    "lm": (上月初, 上月同日),          # 上月同期
    "ly": (去年月初, 去年同日),        # 去年同月
    "ytd": (年初, d),                # 年累计
    "yly": (去年年初, 去年同日),       # 去年年累计
}
```

这种设计的优势是**增加新的对比维度只需增加一个键值对**，而不需要写新的查询逻辑。

### 3.3 数据融合机制

**问题：** API 通常只返回最近 1-2 个月的数据，但年度对比需要至少 12 个月的历史。

**解决方案：** 本地 CSV 增量融合

```
第一次运行：API → CSV（初始建库）
后续每次运行：读 CSV + 读 API → 合并去重 → 回写 CSV
```

核心代码：
```python
combined = pd.concat([csv_df, api_data]).drop_duplicates(subset=['date'], keep='last')
```

`keep='last'` 确保 API 的最新数据优先覆盖本地旧数据。

**关键洞察：** 数据的长期积累是最核心的资产。通过这种融合机制，程序每多运行一天，本地数据库就多积累一天的数据，支撑的分析维度就更丰富。

## 五、关键技术点深挖

### 5.1 同比偏移 (yoy_offset) 的学问

商业分析中，"同比"不是简单的减 365 天：

| 偏移方式 | 天数 | 含义 | 适用场景 |
|---------|------|------|---------|
| 日期对齐 | 365 | 去年的同一个日期 | 月度/年度报告 |
| 星期对齐 | 364 | 去年的同一个星期几 | 日报（避免工作日 vs 周末的偏差） |
| 自定义 | 384 | 特殊业务需求 | 考虑闰年等因素 |

**为什么项目用了 384？** 因为业务方根据实际客流规律，发现 384 天的对比更能反映真实的同比变化，这是运营经验的体现。

### 5.2 周末/平日日均的计算逻辑

不是简单的"总值/天数"，而是要区分**日期类型**：

```python
def get_avg(start, end, dtype):
    # dtype = "周末/假期" 或 "平日"
    mask = (df['dt'] >= start) & (df['dt'] <= end) & (df['date_type'] == dtype)
    sub = df[mask]
    return sub[num_key].sum() / sub.shape[0]
```

**为什么这很重要？** 商场的周末客流可能是平日的 2-3 倍。如果不区分，日均值会被严重扭曲，管理层无法做出正确判断。

### 5.3 条件着色的实现

表格中正负增长的红绿高亮通过 Jinja2 模板条件判断实现：

```html
<td class="{{ 'neg' if '%' in val and '-' in val else ('pos' if '%' in val else '') }}">
```

**设计细节：**
- 只有包含 `%` 的单元格才会着色（避免普通数值被误染）
- 含 `-` 号的为负增长（红色背景 + 红色字体）
- 不含 `-` 号的为正增长（绿色字体，不加背景，避免整片发绿）

### 5.4 Null 值在柱状图中的处理

月均对比柱状图中，今年只有 1-3 月有数据，4-12 月应该留空而不是显示 0：

```python
if m > target_dt.month:
    monthly_cur.append(None)  # None → ECharts 自动跳过
```

如果用 `0` 代替 `None`，ECharts 会画出高度为 0 的柱子（占位但无意义），产生误导。

### 5.5 ECharts animation: false 的必要性

```javascript
const commonOpt = { animation: false, ... };
```

因为最终输出是**静态截图**而非交互式页面，保留动画会导致：
- 截图时图表还在播放动画，导致残缺
- `Html2Image` 的截图时机不确定，结果不稳定

## 六、踩过的坑与解决方案

### 6.1 日期解析警告

**问题：** Pandas 的 `pd.to_datetime()` 对混合格式日期会发出 `UserWarning`

**解决：** 加上 `errors='coerce'` 容错，并在清洗后 `dropna(subset=['dt'])` 过滤掉无法解析的行。

### 6.2 CSV 文件被 Excel 占用

**问题：** 运营人员用 Excel 打开了 `car_history.csv`，程序写入时报 `PermissionError`

**解决：** 
```python
try:
    combined_car.to_csv(csv_file, index=False)
except PermissionError:
    log("⚠️ 无法写入 CSV。请关闭 Excel 后再试。")
```

### 6.3 图片底部被截断

**问题：** 新增图表后，生成的 PNG 底部被截掉

**解决：** 在 `config.ini` 中增大 `height`。坑在于这个值不是图表的实际高度，而是**截图的视口高度**，必须 >= 实际内容高度。

### 6.4 飞书图片上传失败

**问题：** 飞书 Token 有效期约 2 小时，长时间运行后 Token 过期

**解决：** 每次 `generate_report()` 执行时重新创建 `FeishuBot` 实例（重新获取 Token），而不是全局复用。

## 七、可复用的设计模式

### 7.1 "配置外置"模式

**适用场景：** 任何需要在不修改代码的情况下切换参数的项目

```
把"硬编码的魔法值"提取到配置文件 → 代码只读取配置 → 用户改配置即可
```

**Day-Paper 中的实践：**
- 标题前缀 → `config.ini`
- 表格颜色 → `config.ini`
- 行标题 → `config.ini`
- API 地址 → `config.ini`
- 执行时间 → `config.ini`

### 7.2 "字段映射"模式

**适用场景：** 任何需要对接不同数据源的项目

```
定义一个 FIELD_MAP 字典 → 代码中只引用 FIELD_MAP 的 key → 适配新数据只改 value
```

这是一种**依赖倒置**：代码不依赖具体的字段名，而是依赖抽象的语义名。

### 7.3 "引擎+模板"模式

**适用场景：** 任何需要将数据计算和视觉呈现分离的项目

```
计算引擎 (ReportEngine) → 输出纯数据字典 → 模板引擎 (Jinja2) → 渲染为 HTML
```

好处：同一份数据可以渲染成不同风格（暗色模式、简约模式、打印模式等）。

### 7.4 "增量融合"模式

**适用场景：** 任何 API 数据窗口有限但需要长期历史的项目

```
本地存储 + API 增量 → 合并去重 → 回写本地 → 历史越积越多
```

| 自动化思维 | 从手工流程到全链路自动化的转化能力 |

### 9.2 产品思维

| 思维维度 | 具体收获 |
|---------|---------|
| 需求拆解 | 将"做一个日报"拆解为数十个可执行的小任务 |
| 渐进交付 | 先表格 → 再图表 → 最后细节，每步都可验收 |
| 用户视角 | 正负增长着色、周末/平日分开、同比偏移可配 |
| 复用设计 | 从"做一个报表"升级为"做一个报表引擎" |

### 9.3 方法论沉淀

1. **配置驱动 > 代码驱动**：凡是可能变化的值，都不应该写死在代码里
2. **增量积累 > 全量替换**：数据是最核心的资产，本地融合机制让数据越用越多
3. **渐进式开发 > 瀑布式开发**：每一步都能看到结果，驱动下一步优化
4. **引擎化思维 > 脚本化思维**：不是"写一个脚本解决一个问题"，而是"构建一个引擎解决一类问题"

## 十一、总结

Day-Paper 项目从一个"帮运营做日报"的简单需求出发，最终进化成了一套**可配置、可复用、可扩展的经营报表自动化引擎**。

核心启示：

> **不要只解决"这一个问题"，要构建解决"这一类问题"的能力。**

每一行代码、每一个配置项、每一张图表的背后，都是对"如何将手工流程转化为自动化系统"这一命题的深入实践。这种能力迁移到任何领域（财务报表、销售看板、运维监控），核心范式是完全一致的：

```
数据获取 → 计算引擎 → 模板渲染 → 自动分发
```

掌握了这套范式，你就拥有了将任何重复性报表工作自动化的能力。
